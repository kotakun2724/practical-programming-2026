Session

前ページで扱った Cookie は「小さなデータをブラウザに覚えてもらい、リクエストのたびに送ってもらう」仕組みでした。 しかし HTTP 自体はステートレス（各リクエストが独立）なので、「誰が今ログインしているか」のようなサーバー側の状態を直接扱うことはできません。

Session（セッション）は、このギャップを埋めるためのサーバー側でユーザー状態を管理する仕組みです。 前ページの Cookie と組み合わせるのが一般的で、CTF でも頻出のトピックです。

なぜ Session が必要か​

HTTP 単体:
  GET /login   → 「誰?」 (分からない)
  GET /dashboard → 「誰?」 (さっきと同じ? 別人?)

Cookie だけ:
  Cookie: role=admin  → クライアントが書き換え可能。信頼できない。

Session:
  Cookie: session_id=abc123
  サーバー側: { "abc123": { user_id: 1, role: "admin" } }
  → サーバーが状態を持つので、ID だけ分かれば本人と紐付けられる

ポイントは、ブラウザに持たせるのは意味のないランダムな ID だけにして、意味のあるデータ（ユーザーID・権限など）は全部サーバーが持つところです。 これにより、ユーザーは Cookie 値を改竄しても何も得をしません。

サーバーサイドセッションの基本フロー​

1. ユーザーがログインフォームを送信
2. サーバーが認証情報を検証
3. サーバーがランダムなセッションIDを生成（例: 5f4dcc3b5aa765d61d8327deb882cf99）
4. サーバーがセッションストアにIDと対応する状態を保存
     Store: { "5f4dcc3b..." → { user_id: 42, role: "user" } }
5. サーバーが Set-Cookie でセッションIDを返す
     Set-Cookie: session=5f4dcc3b...; HttpOnly; Secure; SameSite=Lax
6. ブラウザが以降のリクエストに自動で Cookie を付ける
7. サーバーは Cookie のセッションIDでストアを引いて、本人確認

実例として、架空のログインエンドポイントに curl でログイン〜 Cookie 保存をしてみます。

terminal
$# 架空の例（example.com には実際の /login エンドポイントや認証情報はない）
$curl -c cookies.txt -sS -L -d "email=user@example.com&password=secret" https://example.com/login -o /dev/null -w "code: %{http_code}\n"
$head -5 cookies.txt
code: 200
# Netscape HTTP Cookie File
# https://curl.se/docs/http-cookies.html
# This file was generated by libcurl! Edit at your own risk.

#HttpOnly_example.com	TRUE	/	TRUE	<有効期限>	session	5f4dcc3b5aa765d61d8327deb882cf99

ログイン成功後、cookies.txt には意味のないランダムなセッションIDだけが保存されています。 このIDが漏れると他人がなりすませるため、HttpOnly + Secure + SameSite=Lax などの Cookie 属性での保護が推奨されます（Cookie の章 参照）。

セッションの実装パターン​

一口に「セッション」と言っても、クライアントに何を渡すかで複数のパターンがあります。

パターンクライアントに保持するもの状態の保管場所主な採用例
サーバーサイドセッション意味のないランダムな セッションIDサーバーストア（メモリ / Redis / DB）PHP, ASP.NET, Java Servlet, Flask（標準）
署名付き Cookieデータ本体（サーバーが署名）クライアント（Cookie 内にすべて格納）Flask の session, Rails の signed cookie, Express の cookie-session
トークン（JWT など）自己完結したトークンクライアント（トークン内） + サーバーは検証のみOAuth 2.0, OpenID Connect, モダン SPA/API

3 つの決定的な違いは、サーバーが状態を持つかどうかです。

  • サーバーサイドセッション → 状態を持つ。失効（ログアウト・BAN）が即時反映できる
  • 署名付き Cookie → 状態は持たない。Cookie 内の署名が本物なら信用する。失効には別テーブル参照が必要
  • JWT → 状態を持たない。署名 + クレーム（exp などの有効期限）で完結。失効が難しい

サーバーサイドセッションの中身​

「セッションストア」と聞くと大掛かりに聞こえますが、突き詰めれば辞書です。

# 概念的な実装（Flask 風）
sessions = {
    "5f4dcc3b5aa765d61d8327deb882cf99": {
        "user_id": 42,
        "role": "user",
        "created_at": 1715000000,
    }
}

# リクエスト受信時
def get_current_user(request):
    sid = request.cookies.get("session")
    if sid not in sessions:
        return None
    return sessions[sid]

実用的には Redis や Memcached などの KVS が使われます。理由:

  • メモリよりプロセス再起動でもデータが残る
  • DB より読み書きが桁違いに速い
  • TTL（有効期限）機能が組み込まれている

ライフサイクル: いつセッションIDを再生成するか​

OWASP の Session Management Cheat Sheet が示す原則は 「特権レベルが変わったら必ず再生成」 です。

タイミング再生成理由
匿名ユーザーがログイン✅ 必須攻撃者が仕込んだ ID を無効化（セッション固定攻撃対策）
ログアウト✅ 必須ストアから削除して即時失効
パスワード変更✅ 必須漏洩疑いのセッションを無効化
権限昇格（一般→管理者）✅ 必須旧権限の Cookie を再利用させない
一定時間アイドル⏰ タイムアウトハイジャックされた場合の被害最小化

「Permissive Session Management」と呼ばれる実装では、ブラウザが送ってきた未知の ID をそのまま受け入れて新規セッションを作るため、攻撃者が事前仕込みした ID を被害者に使わせて乗っ取れます（セッション固定攻撃）。OWASP は Strict な実装（サーバーが生成した ID だけを受け入れる）を推奨しています。

CTF での着眼点​

1. セッションID を確認する​

curl -I https://target/   # Set-Cookie ヘッダーを観察

Set-Cookie: session=5f4dcc3b5aa765d61d8327deb882cf99 のようなランダムな IDなら健全です。 Set-Cookie: session=42 や Set-Cookie: PHPSESSID=1 のような推測可能な値なら、ブルートフォースやインクリメントで突破できるかもしれません。

2. ログイン前後で ID が変わるか確認する​

terminal
$curl -I https://target/profile

ログイン前に匿名セッションID A が発行され、ログイン後も A のままならセッション固定に脆弱な可能性があります。 OWASP の指針通り、ログイン時に 新しい ID へ更新されているべきです。

3. 署名付き Cookie のペイロードを覗く​

Flask の session Cookie は . 区切りで「ペイロード.タイムスタンプ.署名」 の形式です。 ペイロード部分は URL-safe Base64 でエンコードされた zlib 圧縮 JSON なので、デコードできます。

terminal
$python3 -c "
$import base64, zlib
$cookie = 'eJyrVopPy0kszlCyqq7VUUouLkqLL8nPTs1TslJKTEo2NDI2MTVTqgUAACgMUw'
$pad = 4 - len(cookie) % 4
$if pad != 4: cookie += '=' * pad
$raw = base64.urlsafe_b64decode(cookie)
$print('--- zlib 展開前 ---')
$print(repr(raw))
$print('--- zlib 展開後 ---')
$print(zlib.decompress(raw).decode())
$"
--- zlib 展開前 ---
b'x\x9c\xabV\x8aO\xcbI,\xceP\xb2\xaa\xae\xd5QJ..J\x8b/\xc9\xcfN\xcdS\xb2RJLJ642615S\xaa\x05\x00\x00(\x0cS'
--- zlib 展開後 ---
{"_flash":{},"csrf_token":"abc123456"}
警告

Flask の session は itsdangerous で HMAC 署名されています。ペイロードを読めるからといって改竄できるとは限りません。署名検証を通らないとサーバーは無視します。

4. JWT なら alg=none を試す​

JWT の中には alg: none という署名なしトークンを許容してしまう実装ミスがあります。 ヘッダー部分（1 番目の . より前）を Base64 デコードして、alg を確認してみましょう。

{"typ":"JWT","alg":"none"}

alg: none になっていて、署名を空にしても通るなら、ペイロードを自由に改竄できます（次の参考ページで詳述）。

5. Cookie と Session の混同に注意​

「セッション管理しているから安全」と鵜呑みにしてはいけません。 クライアントが改竄できるのか、サーバーで検証しているのかで安全性が大きく違います。

仕組みクライアントが改竄した場合
セッションID別人としてログインし直すだけ（意味ないので攻撃者にとって無益）
署名なし Cookie（role=admin）なりすまし成功（最悪）
署名付き Cookie署名検証で弾かれる（安全）
JWT（署名検証あり）署名検証で弾かれる（安全）

まとめ​

  • セッションは「HTTP をステートフルにする」ための仕組み
  • 主流は サーバーサイドセッション（ランダムなID + サーバーストア）
  • 代替に 署名付き Cookie や JWT もある（クライアント完結型）
  • 特権レベル変更時に必ずセッションID を再生成（セッション固定攻撃対策）
  • セッションID は暗号学的乱数で 64bit 以上のエントロピーが必要（OWASP）
  • CTF では ID の推測可能性、固定化耐性、JWT の alg=none などをチェック
