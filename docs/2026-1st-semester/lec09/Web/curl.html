curlコマンド

curlとは​

curl（Client for URLs）は、コマンドラインからさまざまなプロトコルでリクエストを送信するツールです。 HTTP / HTTPS だけでなく、FTP, SMTP, SCP, SFTP, LDAP, IMAP, MQTT, WebSocket など数十種類のプロトコルに対応しています。

Web CTFでは、ブラウザでは難しいリクエストの細部まで正確に再現・改変できるcurlが必須ツールになります。 今回は HTTP/HTTPS に絞って学びます。

インストール確認​

まずは手元の環境に入っている curl のバージョンを確認しましょう。

terminal
$curl --version
curl 8.20.0 (x86_64-pc-linux-gnu) libcurl/8.20.0 OpenSSL/3.6.2 zlib/1.3.2 brotli/1.2.0 zstd/1.5.7 libidn2/2.3.8 libpsl/0.21.5 libssh2/1.11.1 nghttp2/1.69.0 ngtcp2/1.22.1 nghttp3/1.15.0 mit-krb5/1.21.3
Release-Date: 2026-04-29
Protocols: dict file ftp ftps gopher gophers http https imap imaps ipfs ipns mqtt mqtts pop3 pop3s rtsp scp sftp smtp smtps telnet tftp ws wss
Features: alt-svc AsynchDNS brotli GSS-API HSTS HTTP2 HTTP3 HTTPS-proxy IDN IPv6 Kerberos Largefile libz PSL SPNEGO SSL threadsafe TLS-SRP UnixSockets zstd

バージョン番号、リンクしているライブラリ、対応プロトコル、対応機能（Features）が表示されます。 HTTP/2 や HTTP/3 が使えれば、CTF サーバとの通信でも最新機能が利用できます。

基本的な使い方​

最もシンプルな GET リクエスト​

terminal
$curl https://example.com
<!doctype html><html lang="en"><head><title>Example Domain</title>...<h1>Example Domain</h1><p>This domain is for use in documentation examples without needing permission. Avoid use in operations.</p>...</html>

デフォルトではレスポンスの ボディだけ が出力されます。

進捗表示・エラーメッセージを抑止する -sS​

本番運用ではなく、CTF の探索では静かに動かしたいことがよくあります。 ただし -s だけだと接続エラーも消えるので、-sS (silent + show error) を使うのが定番です。

terminal
$curl -sS https://example.com | head -c 80
<!doctype html><html lang="en"><head><title>Example Domain</title><link rel="icon" hre

レスポンスを見る​

ヘッダーのみ表示: -I / --head​

HTTP レスポンスの ステータスコードやヘッダーだけ を見たいときに使います。 サーバーソフトやキャッシュ情報、隠れたフラグがレスポンスヘッダーに入っている問題は CTF で頻出です。

terminal
$curl -I https://example.com
HTTP/2 200
date: Wed, 17 Jun 2026 13:43:44 GMT
content-type: text/html
server: cloudflare
last-modified: Tue, 16 Jun 2026 20:42:11 GMT
allow: GET, HEAD
accept-ranges: bytes
age: 2878
cf-cache-status: HIT
cf-ray: a0d27c280a234f3d-NRT

ヘッダー + ボディ: -i / --include​

ステータスライン・レスポンスヘッダー・ボディをまとめて出力します。 ブラウザの DevTools の "Network" タブをテキストで再現したイメージです。

terminal
$curl -i https://example.com
HTTP/2 200
date: Wed, 17 Jun 2026 13:43:47 GMT
content-type: text/html
server: cloudflare
last-modified: Tue, 16 Jun 2026 20:42:11 GMT
allow: GET, HEAD

<!doctype html><html lang="en"><head><title>Example Domain</title>...

詳細ログ: -v / --verbose​

TLS ハンドシェイク、HTTP/2 ストリーム、送信ヘッダー、受信ヘッダーまですべて見られます。 「なぜリクエストが届かないのか」 の切り出しに必須です。

-vv でより詳細、-vvv でデバッグレベルまで出力が増えます。

リクエストのメソッド・ヘッダー・ボディ​

メソッドの指定: -X / --request​

CTFでは GET 以外（POST, PUT, DELETE, PATCH など）を使う問題がよく出ます。

terminal
$curl -X POST https://example.com/api

カスタムヘッダー: -H / --header​

Web CTF の最重要オプションです。Cookie、認証トークン、Content-Type、任意のヘッダーを付けられます。

terminal
$curl -H "Content-Type: application/json" -H "X-Forwarded-For: 127.0.0.1" https://example.com/api

-H は 複数回指定できます。@ファイル名 でファイルからヘッダーを読み込むこともできます。

ボディ送信: -d / --data​

terminal
$curl -X POST -d "username=admin&password=hoge" https://example.com/login
terminal
$curl -X POST \
$-H "Content-Type: application/json" \
$-d '{"title":"hello","body":"world","userId":1}' \
$https://jsonplaceholder.typicode.com/posts

実行結果（-i でレスポンスヘッダーを追加）:

terminal
$curl -X POST -H "Content-Type: application/json" -d '{"title":"hello","body":"world","userId":1}' https://jsonplaceholder.typicode.com/posts -i
HTTP/2 201
date: Wed, 17 Jun 2026 13:43:56 GMT
content-type: application/json; charset=utf-8
content-length: 69
location: https://jsonplaceholder.typicode.com/posts/101
access-control-allow-credentials: true
x-powered-by: Express
x-ratelimit-limit: 1000

{
"title": "hello",
"body": "world",
"userId": 1,
"id": 101
}

-d を付けると 自動で Content-Type: application/x-www-form-urlencoded になり、メソッドも POST に切り替わるので、-X POST を明示しなくても同じです。

URL エンコード: --data-urlencode​

日本語や記号を含むクエリを手早く作れます。

terminal
$curl -G \
$--data-urlencode "q=Hello World!" \
$--data-urlencode "lang=日本語" \
$https://example.com/search

-G を付けるとデータをクエリ文字列として URL に付与します。

Cookie とセッション​

CTF のログイン必須問題で必ず使う 3 つのオプションです。

オプション意味例
-b / --cookieCookie を送信（文字列 or ファイル）-b "session=abc123" / -b cookies.txt
-c / --cookie-jar受信した Cookie をファイルに保存-c cookies.txt
-j / --junk-session-cookies-b で読む際にセッション用 Cookie を捨てる-j -b cookies.txt

典型的な流れは次のとおりです。

terminal
$# 1. ログインしてセッション Cookie を取得
$curl -c cookies.txt -d "user=admin&pass=secret" https://target/login
$ 
$# 2. 保存した Cookie を付けてアクセス
$curl -b cookies.txt https://target/dashboard

-c で生成されるファイルは Netscape 形式で、ブラウザや wget と互換性があります。

terminal
$curl -c cookies.txt https://www.google.com/ -o /dev/null
$head -10 cookies.txt
# Netscape HTTP Cookie File
# https://curl.se/docs/http-cookies.html
# This file was generated by libcurl! Edit at your own risk.

#HttpOnly_.google.com	TRUE	/	TRUE	1797255864	__Secure-BUCKET	COoF
#HttpOnly_.google.com	TRUE	/	FALSE	1797515064	NID	532=ZS3i6vYuTzE...
.google.com	TRUE	/	TRUE	1781704164	__Secure-STRP	ANmZwa0Ps9iwd...

リダイレクトと出力​

リダイレクトを追従: -L / --location​

ログイン後や旧 URL から新 URL への 301/302 などを自動で辿ります。

terminal
$curl -L https://example.com/redirect

ボディを保存: -o / --output​

terminal
$curl -o page.html https://example.com
$curl -O https://example.com/path/to/file.zip

進捗・統計を表示: -w / --write-out​

ダウンロードしたサイズや所要時間をフォーマットして出力できます。

terminal
$curl -sS -L -o /dev/null -w "URL: %{url_effective}\nHTTPコード: %{http_code}\nリダイレクト回数: %{num_redirects}\nダウンロードサイズ: %{size_download} bytes\n合計時間: %{time_total} s\n" https://example.com
URL: https://example.com/
HTTPコード: 200
リダイレクト回数: 0
ダウンロードサイズ: 559 bytes
合計時間: 0.091049 s

主な変数:

  • %{http_code}: ステータスコード
  • %{url_effective}: 最終的にアクセスした URL
  • %{num_redirects}: 辿ったリダイレクト数
  • %{size_download}: 受信バイト数
  • %{time_total}: 合計時間（秒）
  • %{content_type}: レスポンスの Content-Type

認証と User-Agent​

Basic 認証: -u / --user​

terminal
$curl -u admin:password https://example.com/admin

User-Agent の指定: -A / --user-agent​

Bot 判定で弾かれるページや、ブラウザ判定で挙動が変わるアプリでは User-Agent の偽装が突破口になります。

terminal
$curl -A "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/126.0.0.0 Safari/537.36" https://example.com

失敗時の挙動​

オプション意味
-f / --failHTTP エラー（4xx, 5xx）が出たらボディを出さずに非ゼロで終了
--fail-with-bodyHTTP エラーでもボディは出すが終了コードは非ゼロ
-S / --show-error-s 併用時でもエラーメッセージを表示
--retry 3一時的な失敗を3回まで再試行
--retry-delay 1再試行の間に1秒待つ
-m 10 / --max-time 1010秒でタイムアウト

シェルスクリプトで扱いやすい書き方:

terminal
$if ! curl -sf https://example.com/health; then
$echo "サーバーが調子悪い"
$exit 1
$fi

HTTP バージョン​

オプション意味
--http1.1HTTP/1.1 を強制
--http2HTTP/2 を要求（ネゴ失敗で 1.1 にフォールバック）
--http2-prior-knowledgeアップグレードなしで HTTP/2 を直接使う
--http3HTTP/3 を要求
--http3-onlyHTTP/3 のみ（失敗したら諦める）

ALPN ネゴシエーションや HTTP/2 の挙動を理解するうえで、curl で明示的にバージョンを固定して試すのは有効です。

CTFでの活用パターン​

1. 隠れたヘッダーを探す​

terminal
$curl -i https://target/

X-Flag-*, X-Admin-*, Server, Set-Cookie あたりは要チェック。

2. メソッドを書き換える​

terminal
$curl -X OPTIONS -i https://target/api
terminal
$curl -X PUT -i https://target/api/user/1

3. アクセス元を偽装する​

terminal
$curl -H "X-Forwarded-For: 127.0.0.1" https://target/admin
terminal
$curl -H "X-Real-IP: 127.0.0.1" https://target/admin

ローカル限定ページを突破できることがあります。

4. ログイン後にリダイレクトを追う​

terminal
$curl -c cookies.txt -L -d "user=admin&pass=guess" https://target/login
$curl -b cookies.txt -L https://target/home

6. セッション Cookie だけ切り捨てる: -j​

ファイルから読み込んだ Cookie のうち、有効期限が設定されていない（セッション中だけ生きる）Cookie だけを捨てるオプションです。 ブラウザを閉じて再起動したのと同じ状態でリクエストを送りたいときに使います。

terminal
$# cookies_test.txt に persistent / session の両方の Cookie が入っていると仮定
$curl -sS -b /tmp/cookies_test.txt -v https://example.com -o /dev/null 2>&1 | grep -i "^> Cookie:"
> Cookie: persistent_cookie=persistent_value; session_cookie=session_value
terminal
$curl -sS -j -b /tmp/cookies_test.txt -v https://example.com -o /dev/null 2>&1 | grep -i "^> Cookie:"
> Cookie: persistent_cookie=persistent_value

上の実行では session_cookie=... が消え、persistent_cookie=... だけが残ります。

5. ワンライナーで総当たり​

terminal
$for path in admin flag.txt robots.txt .env; do
$printf "%s -> " "$path"
$curl -sS -o /dev/null -w "%{http_code}\n" "https://target/$path"
$done

まとめ​

curl はWeb CTF の必須ツールです。最低限、以下を覚えておけば大抵の問題は解けます。

  • -i / -I / -v でレスポンスを覗く
  • -X でメソッドを変える
  • -H でヘッダーを追加・偽装する
  • -d でボディを送る
  • -b / -c で Cookie を扱う
  • -L でリダイレクトを辿る
  • -sS -f -o /dev/null -w "..." でスクリプトに組み込む

迷ったら curl --help <カテゴリ> で使い方を確認する癖をつけましょう。
