Cookie

Cookie（ウェブ Cookie、ブラウザー Cookie）は、サーバーがブラウザーに送信する小さなデータです。 ブラウザーはこれを保存し、同じサーバーに次回以降リクエストする際、自動で送信します。

HTTP プロトコルは既定でステートレス（リクエスト同士が独立）なので、ログイン状態やカートの中身のような「セッション」を実現するために、Cookie がほぼ必須の仕組みとして使われています。

基本的なやりとり​

Cookie は サーバー → ブラウザ と ブラウザ → サーバー の2方向で現れます。

# 1. ブラウザがサーバーに最初のリクエストを送る
GET / HTTP/2
Host: example.com

# 2. サーバーが Set-Cookie で Cookie を返す
HTTP/2 200 OK
Set-Cookie: session_id=abc123; Path=/; HttpOnly; Secure; SameSite=Lax

# 3. 同じサーバーへの以降のリクエストで、ブラウザが自動で Cookie を付ける
GET /dashboard HTTP/2
Host: example.com
Cookie: session_id=abc123
注記

HTTP/2 では Host ヘッダは :authority 擬似ヘッダで運ばれ、レスポンスステータス行もバイナリフレームで表現されます。以下では HTTP/1.1 風に簡略化して書いています。

実例（GitHub に対する curl -I）:

terminal
$curl -sS -I -L https://github.com 2>&1 | grep -i set-cookie
set-cookie: _gh_sess=...; path=/; HttpOnly; secure; SameSite=Lax
set-cookie: _octo=GH1.1....; expires=Thu, 17 Jun 2027 ...; domain=.github.com; path=/; secure; SameSite=Lax
set-cookie: logged_in=no; expires=Thu, 17 Jun 2027 ...; domain=.github.com; path=/; HttpOnly; secure; SameSite=Lax

3本の Set-Cookie が並んでいるのが分かります。Set-Cookie ヘッダは1本のレスポンスに複数並べられます（1本1Cookie）。

Cookie の属性​

Set-Cookie には、Cookie 単体の名前と値に加えて、有効期間・送信条件・JavaScript からの可視性などを制御する属性が付きます。

Set-Cookie: name=value; 属性1; 属性2=値; ...

属性一覧​

属性値効果
ExpiresHTTP-dateCookie の削除日時を GMT で指定。クライアント側の日時と差異が出る可能性があるため、最近は Max-Age が推奨。
Max-Age秒数Cookie の寿命を秒で指定。Expires と両方書かれていれば Max-Age が優先。
Domainドメイン名Cookie をどのホストに送るか。省略時は発行元ホストのみ。サブドメインも対象に含めるには明示的に書く。
PathパスCookie をどの URL パスに送るか。省略時は発行元のパス。/ 区切りで前方一致。
Secure（なし）HTTPS のときだけ Cookie を送信。HTTP では送らない。
HttpOnly（なし）JavaScript からアクセス不可にする（document.cookie 不可）。XSS での窃取を防ぐ。
SameSiteStrict / Lax / Noneサイト間リクエストで Cookie を送るか。CSRF 緩和に効く。
Partitioned（なし）CHIPS（Cookies Having Independent Partitioned State）。サードパーティストレージをパーティションで分離。Secure 必須。
PriorityLow / Medium / High（非標準・Chrome のみ）Cookie 容量超過時の優先度。

Cookie の接頭辞（プレフィクス）​

Cookie 名に __Secure- / __Host- などの接頭辞を付けると、ブラウザーは追加の条件を強制します。 要件を満たさない Cookie は送られても拒否されます。

接頭辞必須条件
__Secure-Secure 属性が付いていること。HTTPS 接続から発行されていること。
__Host-__Secure- の条件 + Domain 属性なし + Path=/
__Http-HTTPS + Secure + HttpOnly すべて必須。Set-Cookie ヘッダー経由でしか設定できない（document.cookie からは作れない）。
__Host-Http-__Host- + __Http- の両方の条件。

GitHub の例で実際に __Secure- 接頭辞が使われているのを見られます:

terminal
$curl -sS -I -L https://github.com 2>&1 | grep -i set-cookie
set-cookie: _gh_sess=...; path=/; HttpOnly; secure; SameSite=Lax
set-cookie: _octo=GH1.1....; expires=...; domain=.github.com; path=/; secure; SameSite=Lax
set-cookie: __Secure-BUCKET=...; ...; Secure; HttpOnly

__Secure-BUCKET にも HttpOnly が付いていますが、これは GitHub が追加で XSS 対策をしているだけで、__Secure- 接頭辞自体は HttpOnly を要求しません（HttpOnly を要求するのは __Http- / __Host-Http- のみ）。

ヒント

CTFで「__Host- 接頭辞付きの Cookie だけ信用できる」前提のアプリを見たときは、ブラウザが Domain 属性の指定を拒否しているおかげで、サブドメインから発行された同名 Cookie が混入しないことが保証されていると分かります。サーバ側でも**値の検証（署名・トークン照合など）**を怠らないようにしましょう。

セッション Cookie vs 永続 Cookie​

Expires と Max-Age の両方とも指定されていない Cookie は、セッション Cookie です。 ブラウザを閉じたとき（セッション終了時）に削除されます。サーバー側は ID を覚えておく必要があるので、本物のセッション管理にはサーバー側のストアが必須です（詳しくは次ページ）。

それ以外（Expires または Max-Age が指定されている）は永続 Cookieで、ブラウザを閉じても残り、有効期限まで送信されます。

種類例
セッション CookieSet-Cookie: session_id=abc123; Path=/; HttpOnly
永続 CookieSet-Cookie: theme=dark; Max-Age=2592000; Path=/ （30日間）
警告

多くのブラウザーにはセッション復元機能があり、ブラウザを閉じて再起動してもセッション Cookie が生き残ることがあります。テスト時には意識しておく必要があります。

サイズと数の上限​

RFC 6265 は「Cookie 1個あたり最低 4096 バイトまで対応すべき」と定めています。 実ブラウザーは概ねこのあたりに収まり、おおむね:

  • 1 Cookie あたり ~4KB
  • 1 ドメインあたり 数百個

を超えると古いものから破棄されたり、保存自体が拒否されたりします。 Cookie に大きなデータを入れたい誘惑に駆られますが、リクエストごとに自動送信されるため帯域・パフォーマンス両面で悪影響です。 大きなデータは IndexedDB / Web Storage を使いましょう。

SameSite の挙動まとめ​

値同じサイト内別サイト → 普通に遷移別サイト → fetch/img/script別サイト → POST
Strict✅ 送る❌ 送らない❌ 送らない❌ 送らない
Lax✅ 送る✅ 送る❌ 送らない❌ 送らない（一部例外あり）
None（Secure 必須）✅ 送る✅ 送る✅ 送る✅ 送る

SameSite 属性が指定されていない modern ブラウザーの既定値は Lax です。

ブラウザでの確認方法​

Chrome / Edge の DevTools​

  1. F12 で DevTools を開く
  2. Application タブ → 左サイドバーの Storage > Cookies を選択
  3. ドメインごとの Cookie 一覧（名前、値、Domain、Path、Expires、Size、HttpOnly、Secure、SameSite）が表示される

Firefox​

  1. F12 で DevTools を開く
  2. ストレージタブ → Cookie を選択
  3. 同じく一覧表示

コンソールからのクイック確認​

// JavaScript からは HttpOnly 以外の Cookie が見える
console.log(document.cookie);
// 例: "theme=dark; lang=ja"

HttpOnly の Cookie は document.cookie にも出てきません。curl -I や DevTools の Cookies パネルでないと確認できません。

CTF での着眼点​

1. Set-Cookie を必ず確認する​

flag が Cookie 値そのものに入っている問題は CTF の入門で頻出です。

curl -I https://target/   # ヘッダーだけ
curl -i https://target/   # ヘッダー + ボディ

2. 権限・状態を表す Cookie 名に着目する​

  • session, PHPSESSID, JSESSIONID, connect.sid … セッションID
  • role, admin, is_admin, user_id … 権限・本人特定
  • lang, theme, currency … 表示設定
  • eyJ... 形式 … Base64 (URL-safe) の JSON。デコードできる＝改竄のリスクがある。

role=user を role=admin に書き換えるだけで突破できる問題は典型例です。

3. HttpOnly じゃない Cookie は改竄できる​

HttpOnly 属性がない Cookie は、ブラウザの DevTools で値を直接書き換えられます。 サーバー側で値の検証（署名・DB引き直し）をしていないアプリは突破できます。

# curl で任意の Cookie を付ける
curl https://target/admin -b "role=admin; user_id=1"

4. 署名付き Cookie の構造を読む​

Flask の session Cookie や JWT は、ヘッダー.ペイロード.署名 の形式です。 ペイロード部分は Base64 デコードすれば中身が読めます（次のセッションで詳述）。

5. SameSite=None + Secure の意味​

SameSite=None の Cookie はサードパーティ Cookieとして動作します。 最近のブラウザーはサードパーティ Cookie をブロックする方向なので、SameSite=None; Secure がないと CTF プラットフォーム上でログインセッションが維持できないことがあります。AlpacaHack 等で「ログイン状態が切れる」現象に出くわしたら、まずここを疑います。

まとめ​

  • Cookie は HTTP をステートフルにする仕組みで、サーバー発行の Set-Cookie と、ブラウザが自動で付ける Cookie ヘッダーの往復で動く
  • 属性（Expires/Max-Age, Domain, Path, Secure, HttpOnly, SameSite, Partitioned, 接頭辞）で寿命・送信条件・可視性を細かく制御する
  • サイズ・数に上限がある（1個 ~4KB、1ドメイン数百個）
  • セッション Cookie はブラウザ終了で消える、永続 Cookie は期限まで残る
  • CTF では Set-Cookie の観察、Cookie 値の Base64 デコード、HttpOnly でない値の改竄が基本テク

詳細な属性リファレンス・仕様・ブラウザ実装の差は 付録: Cookie 詳細リファレンス にまとめます。
