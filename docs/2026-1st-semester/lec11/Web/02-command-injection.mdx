# Command Injection

## 今回の目標

1. **Command Injection**の仕組みを理解する
2. **コマンド区切り文字**（`;`, `&&`, `|`, `` ` ``, `$()`）を使い分ける
3. 防御の基本（OSコマンドを呼ばない・厳密な入力検証）を理解する

## Command Injectionとは

**Command Injection（コマンドインジェクション）** は、Webアプリケーションへの攻撃手法の一つです。入力欄に不正なOSコマンドを注入し、サーバー上で任意のコマンドを実行させます（**RCE: Remote Code Execution**）。

### 前提条件

この攻撃が成立するには：

- WebアプリケーションがOSコマンドを実行している（`ping`, `traceroute`, `curl`, 画像処理等）
- 入力値がコマンドに文字列結合されている
- 適切な入力検証が行われていない

:::note サーバーのOSで差が出る点

使える **区切り文字** や **コマンド名** がサーバーのOSによって異なります。

| OS | シェル | よく使う区切り文字 |
|---|---|---|
| Linux | sh / bash | `;`, `&&`, `\|`, `` ` ``, `$()` |
| Windows | cmd.exe | `&`, `&&`, `\|`, `^` |

CTF のサーバーは Linux が多いですが、入力が Windows 向けにサニタイズされている場合は挙動が変わります。

:::

### 攻撃成功/失敗の判定

- **成功**: コマンド実行結果がレスポンスに含まれる、またはflagファイルの内容が表示される
- **失敗**: 入力検証エラー、またはコマンド実行エラー

## 攻撃の仕組み

### 脆弱なコード（擬似コード）

```
# ユーザー入力を使ってpingを実行
hostname = 入力
実行("ping -c 4 " + hostname)
```

### 攻撃例

```
入力: 8.8.8.8; cat /etc/passwd
```

### 実行されるコマンド

```bash
ping -c 4 8.8.8.8; cat /etc/passwd
```

`;` でコマンドが区切られ、`cat /etc/passwd` が実行されます。

## コマンドの区切り文字

| 文字 | 説明 | 例 |
|---|---|---|
| `;` | コマンドを順に実行 | `cmd1; cmd2` |
| `&&` | 前のコマンドが成功したら実行 | `cmd1 && cmd2` |
| `\|\|` | 前のコマンドが失敗したら実行 | `cmd1 \|\| cmd2` |
| `\|` | パイプ（前の出力を次の入力に） | `cmd1 \| cmd2` |
| `` ` `` | バッククォート（コマンド置換） | `` cmd1 `cmd2` `` |
| `$()` | コマンド置換 | `cmd1 $(cmd2)` |

Windows 系ではさらに `&`（`&&` と同義）がよく使われます。

## 実践: Command Injection

### 基本的なペイロード

```
; ls
; cat /etc/passwd
; id
; whoami
```

### バッククォートや$()を使う

```
`ls`
$(ls)
```

## よくある脆弱性パターン

### ネットワークツールの入力欄

- `ping`
- `traceroute`
- `nslookup`

### ファイル名にコマンドを埋め込む

```
# 擬似コード
ファイル名 = 入力
実行("cat /uploads/" + ファイル名)
```

```
入力: $(cat /etc/passwd)
```

バッククォートや`$()`を使うと、ファイル名の入力欄でもコマンドを実行できます。

### curl / fetch を「サービス」として提供している

`curl <ユーザー入力>` を exec するようなアプリでは、`-o` で任意ファイルに書き出したり、`scp://`・`sftp://` など別プロトコルのURLを渡したりできる点に注意。本編で扱う `;` 区切りによる古典的コマンド注入だけでなく、**引数インジェクション** の可能性も意識しましょう。

## 防御方法

### 1. OSコマンドを実行しない

最も安全な方法は、OSコマンドを実行しないことです。代替として、各言語のライブラリ（Pythonの `socket`、Goの `net` 等）を使いましょう。

### 2. 入力値の厳密な検証

やむを得ずコマンドを実行する場合は、入力をホワイトリストで検証します。

```
# 英数字とドット・ハイフンのみ許可
もし 入力 が [a-zA-Z0-9.-] にマッチしないなら:
    エラー
```

### 3. シェルを経由しない

`system()` や `os.system` は **シェル経由** で実行されるため、区切り文字がすべて解釈されてしまいます。`execve` や `subprocess.run(..., shell=False)` のように、**引数を配列で渡し、シェルを介さない** 実行方法を使いましょう。

:::note 防御側の視点

### 検出方法

- OSコマンドの実行ログ
- 不審なプロセス起動の検出
- WAFによるコマンドパターンのブロック
- ファイルシステムの変更監視

### 攻撃成功時のレスポンス例

```http
HTTP/1.1 200 OK
Content-Type: text/plain

PING 8.8.8.8 (8.8.8.8): 56 data bytes
64 bytes from 8.8.8.8: icmp_seq=0 ttl=117 time=10.5 ms
FLAG{command_injection_success}
```

:::

## CTFでのポイント

1. **入力欄を全て試す**: 特にファイル名、ホスト名、コマンド関連
2. **様々な区切り文字を試す**: `;`, `&&`, `|`, `` ` ``, `$()`
3. **flagの場所を探す**: `/flag.txt`, `/home/ctf/flag`, 環境変数など

## よく使うコマンド

| コマンド | 用途 |
|---|---|
| `id`, `whoami` | 現在のユーザー確認 |
| `ls` | ファイル一覧 |
| `cat` | ファイル内容表示 |
| `find` | ファイル検索 |
| `env` | 環境変数表示 |

## まとめ

Command Injectionは、サーバーの完全な制御を許す非常に危険な攻撃手法です。OSコマンドの実行は避け、やむを得ない場合は厳密な入力検証と **シェルを経由しない実行** が必要です。
