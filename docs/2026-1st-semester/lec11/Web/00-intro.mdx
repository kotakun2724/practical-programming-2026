# はじめに

:::warning

ここで学ぶ技術は， **CTFの範囲内でのみ** 使用してください．許可なく他人のシステムを攻撃することは **不正アクセス禁止法** で禁じられています．演習問題は必ず AlpacaHack などの公式プラットフォーム上で解いてください．

:::

## 今回の目標

前回は Web CTF の **リクエスト改ざん** を学びました。今回は一歩進んで、ユーザー入力を **そのままシステムに流してしまう** 脆弱性を突く **Injection 攻撃** を扱います。

1. **SQL Injection** — ログインフォームや検索欄に SQL 文を注入し、データベースの情報を不正に取得する
2. **Command Injection** — 入力欄に OS コマンドを注入し、サーバー上で任意のコマンドを実行する（RCE）
3. 防御の基本 — **パラメータ化クエリ** / **エスケープ** / **入力検証** / **OSコマンドを呼ばない**

## 前回のおさらい

前回（lec10）では、**「クライアントから送られる値は、全部書き換えられる」** という大前提のもと、URL/フォーム/Cookie/ヘッダーを改ざんしてサーバー側のロジック穴を突く攻撃（ディレクトリトラバーサル、IP制限回避、プロトタイプチェーン悪用など）を学びました。

## 今回の立ち位置

> **ユーザー入力を「そのまま」システムに流してはいけない**

SQL Injection も Command Injection も根本原因は同じです。**信頼できない入力を、SQL 構文やシェルコマンドの構文の一部として結合してしまう** ことで、攻撃者が構文を自由に組み立てられる状態になります。

| | 標的 | 結合される構文 |
|---|---|---|
| SQL Injection | DB（データベース） | SQL 文 |
| Command Injection | OS（サーバーのシェル） | シェルコマンド |

どちらも **入力と構文を分離する** ことで完全に防御できます。次ページから具体的な攻撃手法を見ていきましょう。
