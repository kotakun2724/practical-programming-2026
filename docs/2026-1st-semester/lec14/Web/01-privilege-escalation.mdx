# 特権昇格

## 今回の目標

1. **特権昇格（Privilege Escalation）** の仕組みを理解する
2. **水平権限昇格** と **垂直権限昇格** の違いを理解する
3. 認可チェックの不備を突く代表的な手法を、これまでの知識と結びつけて整理する

## 特権昇格とは

**特権昇格（Privilege Escalation）** は、自分に割り当てられた権限を超えて、より高い権限や他人の権限を不正に取得する攻撃手法です。大きく2種類に分かれます。

| 種類 | 内容 | 例 |
|---|---|---|
| **水平権限昇格**（Horizontal） | **同じ権限レベルの他ユーザー** のリソースにアクセスする | 一般ユーザー A が一般ユーザー B のプロフィールを閲覧する |
| **垂直権限昇格**（Vertical） | **より上位の権限** を取得する | 一般ユーザーが管理者ページにアクセスする |

lec10 で「認可の不備」として扱った内容は、実はこの特権昇格の一部でした。今回はそれを **権限の上下関係** という視点で整理し直します。

### 前提条件

この攻撃が成立するには：

- 認可チェックが、クライアントから送られる値（Cookie・パラメータ・JWT のクレーム）に依存している
- サーバー側で「その操作をする権限があるか」の検証が不足している

### 攻撃成功/失敗の判定

- **成功**: 一般ユーザーの資格情報で管理者向け API・画面にアクセスできる、他人のリソースが参照できる
- **失敗**: `403 Forbidden`、トップページへのリダイレクト

## 攻撃手法

### 1. IDOR（水平権限昇格）

**IDOR（Insecure Direct Object Reference）** は、オブジェクトへの直接参照が安全でない脆弱性です。

```
/profile?id=123  →  /profile?id=1   （他人のプロフィールが見える）
```

lec10 の「認可の不備」で扱った例そのものです。ID だけでなく、ファイル名・注文番号・UUID など、**リソースを指す値全般** が対象になります。「ID を知っていれば誰でもアクセスできる」設計が根本原因です。

### 2. 強制ブラウジング（垂直権限昇格）

**強制ブラウジング（Forced Browsing）** は、リンクされていない URL を直接指定して、隠れたページにアクセスする手法です。

```
/user/profile  →  /admin/profile
```

管理者ページへのリンクが画面に出ていなくても、URL を直接叩けば届きます。「ページが隠れている＝安全」ではありません。

### 3. パラメータ改竄・マスアサインメント（垂直権限昇格）

lec10 で扱ったパラメータ改竄の応用です。プロフィール更新 API が権限フィールドまで受け付けてしまっていると、自分の権限を書き換えられます。

```http
POST /profile HTTP/1.1
Content-Type: application/json

{"user_id": 123, "role": "user"}
```

を

```http
{"user_id": 123, "role": "admin"}
```

に変更します。サーバーがリクエストの全フィールドを無検証で DB に反映する実装（**マスアサインメント**）だと、本来いじれないはずの `role` まで更新できてしまいます。

### 4. JWT の改竄（垂直権限昇格）

lec13 で扱った JWT 改竄も、特権昇格の一種です。

```json
{"user_id": 123, "role": "user"}
```

を

```json
{"user_id": 123, "role": "admin"}
```

に改竄します。`alg=none` が許容される、または秘密鍵が弱くて署名を再生成できる場合に成立します。

## 攻撃チェーンとしての特権昇格

これまでの講義で学んだ攻撃は、単独ではなく **連鎖（チェーン）** させて使うのが実戦的です。

```
1. NoSQL Injection で認証バイパス（lec13）
   → 一般ユーザーとしてログイン成功

2. JWT の alg=none で role を admin に改竄（lec13・今回）
   → 管理者権限を取得

3. 管理者向け API から flag を取得
```

CTF の Medium 以上の問題では、このように **複数の小さな不備を繋いで** flag にたどり着く構成が一般的です。1つの脆弱性が見つかっても「それだけでは flag に届かない」とき、次に繋げられる攻撃がないか考えるのが攻略の鍵です。

## 防御方法

### 1. サーバー側での認可チェック

リクエストごとに「現在のユーザーがその操作を行う権限があるか」を **必ずサーバー側で** 確認します。デフォルトは **拒否** とし、許可された操作だけを通す設計にします。

```
# 擬似コード
もし 現在のユーザー.role != "admin" なら:
    403 Forbidden を返す
```

### 2. オブジェクトごとの所有権チェック

`/profile?id=1` のような参照では、「そのリソースの **所有者** が現在のユーザーか」を検証します。ID をクライアントから直接受け取らず、サーバー側でセッションから参照先を決める **間接参照** も有効です。

### 3. セッション情報はサーバー側を信頼する

```
# 悪い例: クライアントが送る値を信用している
もし Cookieの "role" == "admin" なら:
    許可

# 良い例: サーバー側のセッションストアを参照する
もし セッション（サーバー側）の "role" == "admin" なら:
    許可
```

Cookie・パラメータ・JWT のクレームなど、**クライアントから届く値はすべて改竄可能**（lec10 の大前提）です。権限の判断材料はサーバー側で管理します。

:::note

### 検出方法

- 一般ユーザーのセッションで管理者向けエンドポイントへのアクセスログ
- `role` / `is_admin` などの権限フィールドを含む不審なリクエストボディ
- 他人のリソース ID への連続アクセス（IDOR の探索行動）

### 攻撃成功時のレスポンス例

```http
HTTP/1.1 200 OK
Content-Type: application/json

{"status":"ok","role":"admin","flag":"FLAG{privilege_escalation_success}"}
```

:::

## CTFでのポイント

1. **全てのパラメータを観察**: `role`, `admin`, `is_admin`, `user_id`
2. **URL を操作**: `/user/` → `/admin/`、隠れたエンドポイント（`/admin`, `/api/admin`, `/internal`）
3. **JWT をデコード**: `role` を `admin` に変更して再署名できるか
4. **水平と垂直を意識する**: 「他人のものが見えるか」（水平）と「上の権限に上がれるか」（垂直）は別の穴

## まとめ

特権昇格は、認可の不備を突いて権限を不正に拡大する攻撃です。**水平**（他人のリソース）と **垂直**（上位権限）の2軸で整理すると、これまで学んだ IDOR・パラメータ改竄・JWT 改竄がすべてこの枠組みに収まります。防御の要は **サーバー側での認可チェック** です。
