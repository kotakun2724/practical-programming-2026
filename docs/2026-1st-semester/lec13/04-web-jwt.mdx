# JWT（JSON Web Token）

## JWTとは

**JWT（JSON Web Token）**は、JSONオブジェクトとして情報を安全に転送するためのトークン形式です。

Webアプリケーションで、認証によく使われます。

## JWTの構造

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyX2lkIjoxLCJyb2xlIjoiYWRtaW4ifQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQsW5c
```

JWTは3つの部分から成り立ちます：

```
[Header].[Payload].[Signature]
```

### 1. Header（ヘッダー）

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

- `alg`: 署名アルゴリズム（HS256など）
- `typ`: トークンのタイプ

### 2. Payload（ペイロード）

```json
{
  "user_id": 1,
  "role": "admin",
  "exp": 1234567890
}
```

- `user_id`, `role`: ユーザー情報
- `exp`: 有効期限

### 3. Signature（署名）

```
HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret_key
)
```

署名により、トークンの改竄を検出できます。

## Base64Urlエンコード

JWTの各部分は**Base64Urlエンコード**されています。

### デコード例

```bash
echo "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9" | base64 -d
# {"alg":"HS256","typ":"JWT"}

echo "eyJ1c2VyX2lkIjoxLCJyb2xlIjoiYWRtaW4ifQ" | base64 -d
# {"user_id":1,"role":"admin"}
```

### jwt.io

Webサイト [https://jwt.io](https://jwt.io) でJWTのデコード・検証ができます。

## JWTの脆弱性

### 1. 署名の検証なし（Noneアルゴリズム）

```json
{
  "alg": "none"
}
```

署名アルゴリズムを `none` に設定することで、署名の検証をスキップできる脆弱性があります。

#### 攻撃例

```bash
# Headerを {"alg":"none","typ":"JWT"} に変更
echo -n '{"alg":"none","typ":"JWT"}' | base64
# eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0

# Payloadを {"user_id":1,"role":"admin"} に変更
echo -n '{"user_id":1,"role":"admin"}' | base64
# eyJ1c2VyX2lkIjoxLCJyb2xlIjoiYWRtaW4ifQ

# 署名なしのJWTを作成
# eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJ1c2VyX2lkIjoxLCJyb2xlIjoiYWRtaW4ifQ.
```

### 2. 弱い秘密鍵

```bash
# 一般的なパスワードで署名を検証してみる
# jwt_toolなどのツールを使用
```

秘密鍵が `secret`, `password`, `key` などの一般的な場合、推測できます。

### 3. ペイロードの改竄

署名が正しく検証されていない場合、ペイロードを改竄できます。

```json
{
  "user_id": 1,
  "role": "user"
}
```

を

```json
{
  "user_id": 1,
  "role": "admin"
}
```

に変更できます。

## CTFでのポイント

1. **JWTをデコードする**: ペイロードの内容を確認
2. **署名アルゴリズムを確認**: `none` に変更できるか試す
3. **秘密鍵を推測**: 一般的なパスワード
4. **ペイロードを改竄**: `role` を `admin` に変更
5. **有効期限を確認**: `exp` クレームの検証

## 防御方法

### 1. 署名の検証

必ず署名を検証します。

### 2. 強い秘密鍵

```
secret_key = 十分な長さのランダムな文字列
```

### 3. アルゴリズムの固定

```
許可するアルゴリズム = ["HS256"] のみ
```

`none` や他のアルゴリズムを許可しません。

## まとめ

JWTは便利なトークン形式ですが、実装の不備により様々な脆弱性が生まれます。

署名の検証、強い秘密鍵、適切なアルゴリズムの使用が重要です。

次回は、認証応用（特権昇格、セッションハイジャック）について学びます。
