# 認証の基礎

## 前回の振り返り

前回の演習（Alert my Flag）では、`eval`と文字列連結でフィルターをバイパスするXSSを学びました。今回は、Webアプリケーションの認証・セッション管理を学びます。

## 今回の目標

1. **認証**と**認可**の違いを理解する
2. **Cookie**と**Session**の仕組みを理解する
3. **JWT**の構造と脆弱性を学ぶ

## 認証（Authentication）と認可（Authorization）

### 認証（Authentication）

「あなたは誰ですか？」を確認するプロセスです。

- ユーザー名とパスワード
- ワンタイムパスワード

### 認可（Authorization）

「何をする権限がありますか？」を確認するプロセスです。

- 管理者ページのアクセス権
- ファイルの読み書き権限

## 認証方式

### 1. Basic認証

```
Authorization: Basic dXNlcjpwYXNzd29yZA==
```

`dXNlcjpwYXNzd29yZA==` は `user:password` をBase64エンコード（データを文字列に変換する方式）したものです。

:::warning
Base64はエンコードであり、暗号化ではありません。簡単にデコードされるため、HTTPSと併用する必要があります。
:::

### 2. Form-Based認証

HTMLフォームを使ってログインします。

```html
<form method="POST" action="/login">
  <input type="text" name="username">
  <input type="password" name="password">
  <button type="submit">ログイン</button>
</form>
```

### 3. トークン認証

サーバーが発行したトークンをリクエストに含めます。

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

## セッション管理

### 1. Cookieベースのセッション

```
Set-Cookie: session_id=abc123; HttpOnly; Secure
```

サーバー側でセッション情報を管理し、CookieでセッションIDをやり取りします。

### 2. トークンベースのセッション

```json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

JWTなどのトークンを使って、クライアント側で状態を管理します。

## よくある脆弱性

### 1. 弱いパスワード

```
admin / password
admin / admin123
```

### 2. パスワードのブルートフォース

```bash
curl -X POST https://target.com/login \
  -d "username=admin&password=123456"

curl -X POST https://target.com/login \
  -d "username=admin&password=password"
```

### 3. セッションIDの推測

```
session_id=1
session_id=2
session_id=3
```

セッションIDが連番で生成されている場合、他のユーザーのセッションを推測できます。

### 4. NoSQL Injection

MongoDBなどのNoSQLデータベースでは、JSON形式でクエリを送信できる場合があります。

```javascript
// サーバー側のコード（Node.js + MongoDB）
db.users.findOne({ username: username, password: password })
```

ここで、JSONで `{"username": "admin", "password": {"$ne": ""}}` を送信すると：

```javascript
// 展開後のクエリ
db.users.findOne({ username: "admin", password: { $ne: "" } })
// → passwordが空文字でないadminユーザーにマッチ
```

`$ne` はMongoDBの演算子で「等しくない」を意味します。これにより、パスワードを知らずに認証をバイパスできます。

## 防御方法

### 1. 強いパスワードポリシー

- 最低8文字以上
- 大文字・小文字・数字・記号を含む

### 2. アカウントロック

```
5回連続ログイン失敗 → 30分間ロック
```

### 3. セッション管理

- `HttpOnly` フラグでJavaScriptからのアクセスを禁止
- `Secure` フラグでHTTPSのみで送信
- 適切な有効期限の設定

## CTFでのポイント

1. **デフォルトの認証情報**を試す: `admin/admin`, `admin/password`
2. **ブルートフォース**を試す: 弱いパスワード
3. **セッションID**を観察する: 推測可能なパターン
4. **認証バイパス**を試す: URLの操作、パラメータの改竄

## まとめ

認証とセッション管理は、Webセキュリティの重要な要素です。

弱い実装は、認証バイパスやセッションハイジャックの原因となります。

次は、**JWT（JSON Web Token）**について詳しく学びます。JWTはトークンベースの認証で使われる仕組みで、適切に実装しないと署名のバイパスやペイロードの改竄を許す可能性があります。
