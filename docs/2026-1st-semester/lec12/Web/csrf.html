CSRF（Cross-Site Request Forgery）

今回の目標​

  1. CSRFの仕組みと危険性を理解する
  2. XSSとの違いを理解する
  3. 防御の基本（CSRFトークン / SameSite Cookie）を理解する

CSRFとは​

CSRF（Cross-Site Request Forgery） は、Webアプリケーションへの攻撃手法の一つです。認証済みのユーザーに、意図しないリクエストを送信させます。

ユーザーが攻撃者の罠ページにアクセスすることで、知らないうちに危険な操作（パスワード変更、送金など）が実行されます。

前提条件​

この攻撃が成立するには：

  • 被害者が認証済みである（CookieにセッションIDが含まれている）
  • CSRFトークンが設定されていない
  • SameSite属性が設定されていないCookieを使用している
  • ブラウザが自動的にCookieを付与する仕様（これが根本原因）

攻撃成功/失敗の判定​

  • 成功: 意図しない操作が実行される（送金完了、パスワード変更完了等）
  • 失敗: CSRFトークン検証エラー、403 Forbidden

XSSとの違い​

XSSCSRF
何ができるか任意のJavaScriptを実行認証済みユーザーのリクエストを送信
対象他のユーザーのブラウザWebアプリケーション
主な目的Cookie窃取、情報窃取意図しない操作の実行

攻撃の仕組み​

攻撃の流れ​

1. ユーザーが銀行サイトにログイン（CookieにセッションIDが保存される）
2. ユーザーが攻撃者の罠ページにアクセス
3. 罠ページが、銀行サイトへのリクエストを自動送信
4. ブラウザは自動的にCookie（セッションID）を付与
5. 銀行サイトは「正規ユーザーのリクエスト」と認識して処理を実行

攻撃例​

罠ページ（攻撃者のサイト）​

<!DOCTYPE html>
<html>
<body>
  <h1>無料の景品ゲット！</h1>
  
  <form id="csrf" action="https://bank.com/transfer" method="POST">
    <input type="hidden" name="to" value="attacker_account">
    <input type="hidden" name="amount" value="1000000">
  </form>
  
  <script>
    document.getElementById('csrf').submit();
  </script>
</body>
</html>

CSRFの種類​

1. GETベースのCSRF​

<img src="https://bank.com/transfer?to=attacker&amount=1000000" width="0" height="0">

2. POSTベースのCSRF​

<form action="https://bank.com/transfer" method="POST" id="csrf">
  <input type="hidden" name="to" value="attacker">
  <input type="hidden" name="amount" value="1000000">
</form>
<script>
  document.getElementById('csrf').submit();
</script>

防御方法​

1. CSRFトークン​

<form action="/transfer" method="POST">
  <input type="hidden" name="csrf_token" value="random_token_12345">
  <input type="text" name="to">
  <input type="number" name="amount">
</form>

サーバー側でトークンを検証します。攻撃者はトークンを知らないため、偽のリクエストを送信できません。

2. SameSite Cookie​

Set-Cookie: session=abc123; SameSite=Strict

SameSite属性は、クロスサイトリクエストでCookieを送信するかどうか を制御します。

値動作CSRF防御
Strict同一サイトからのリクエストのみCookie送信完全に防御
LaxGETリクエストのみクロスサイトでも送信（POSTはブロック）ほぼ防御（POST CSRFは防げる）
Noneクロスサイトでも常に送信（Secure属性必須）防御しない

モダンブラウザでは Lax がデフォルトです。SameSiteを指定しない場合でも、Lax 相当の動作になります。

なぜLaxでPOST CSRFが防げるのか​

攻撃者の罠ページ（https://attacker.com）から <form method="POST"> を送信しても、SameSite=Lax のCookieは付与されません。一方、<a> タグや <img> タグによるGETリクエストではCookieが送信されるため、GETベースのCSRFには注意が必要です。

実装例​

# 悪い例: SameSiteなし（Laxがデフォルトだが、明示すべき）
Set-Cookie: session=abc123

# 良い例: 明示的にLaxを指定
Set-Cookie: session=abc123; SameSite=Lax

# より安全: Strict（ただしUXに影響あり）
Set-Cookie: session=abc123; SameSite=Strict
注記

SameSite=Strict は、外部リンク（例: Google検索結果）からサイトにアクセスした際にもCookieが送信されないため、ログイン状態が維持されない というUX上の問題があります。多くのサイトでは Lax を使用しています。

注記

検出方法​

  • CSRFトークンの検証失敗ログ
  • 不審なRefererヘッダーの検出
  • ユーザーの操作パターン分析
  • 重要な操作前の再認証

攻撃成功時のレスポンス例​

HTTP/1.1 200 OK
Content-Type: application/json

{"status": "Transfer completed", "to": "attacker", "amount": 1000000}

CTFでのポイント​

  1. CSRFトークンの確認: フォームにCSRFトークンがあるか確認する
  2. Cookieの属性: SameSite 属性が設定されているか確認する
  3. 管理者の操作を模倣: 管理者だけが実行できる操作（flagの表示など）を、CSRFで実行させる
  4. Botの動作: CTFでは、管理者Botが罠ページにアクセスする設定になっていることがある

まとめ​

CSRFは、認証済みのユーザーに意図しない操作を実行させる攻撃手法です。CSRFトークンとSameSite Cookieの設定で防御可能です。
