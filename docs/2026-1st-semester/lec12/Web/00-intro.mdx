# はじめに

:::warning

ここで学ぶ技術は， **CTFの範囲内でのみ** 使用してください．許可なく他人のシステムを攻撃することは **不正アクセス禁止法** で禁じられています．演習問題は必ず AlpacaHack などの公式プラットフォーム上で解いてください．

:::

## 今回の目標

前回は Web CTF の **Injection 攻撃**（SQL Injection, Command Injection）を学びました。今回は一歩進んで、**クライアント側のブラウザ** を標的とする攻撃を扱います。

1. **XSS（Cross-Site Scripting）** — Webページにスクリプトを注入し、他のユーザーのブラウザで実行させる
2. **CSRF（Cross-Site Request Forgery）** — 認証済みユーザーに、意図しないリクエストを送信させる
3. 防御の基本 — **出力のエスケープ** / **CSP** / **CSRFトークン** / **SameSite Cookie**

## 前回のおさらい

前回（lec11）では、**「ユーザー入力をそのままシステムに流してしまう」** 脆弱性を突く **Injection 攻撃** を学びました。

| 攻撃 | 標的 | 結合される構文 |
|---|---|---|
| SQL Injection | DB（データベース） | SQL 文 |
| Command Injection | OS（サーバーのシェル） | シェルコマンド |

どちらも **サーバー側** のシステムを攻撃する手法でした。

## 今回の立ち位置

> **Injection 攻撃はサーバー側を標的にしたが、XSS/CSRF はクライアント側を標的にする**

XSS も CSRF も、**他のユーザーのブラウザ** を攻撃します。サーバーではなく、**サーバーを利用しているユーザー** が標的です。

| | 標的 | 何ができるか |
|---|---|---|
| XSS | 他のユーザーのブラウザ | 任意のJavaScriptを実行 |
| CSRF | Webアプリケーション（ユーザーの権限を悪用） | 認証済みユーザーのリクエストを送信 |

どちらも **信頼できない入力やリクエストを、そのまま処理してしまう** ことが根本原因です。次ページから具体的な攻撃手法を見ていきましょう。
